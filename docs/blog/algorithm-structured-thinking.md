# 算法竞赛中的结构化思维

做题时最容易浪费时间的不是不会写代码，而是思路没有及时收束。对我来说，更有效的方法不是“多猜几个做法”，而是把题目拆进一套固定框架里。

## 我常用的四步

1. 先确认对象：题目究竟在限制什么，是结构、顺序、状态还是代价。
2. 再找变量：哪些量在变化，哪些量可以预处理，哪些量必须在线维护。
3. 判断复杂度：枚举一层还是两层，是否需要数据结构接管。
4. 最后才选算法：图论、DP、贪心还是构造，不要反过来。

## 为什么这套方法有效

它能减少被题面干扰的概率。很多题看起来新，其实只是包装不同；一旦把问题抽回到“状态如何转移”或者“顺序如何维护”，难度就会明显下降。

## 接下来准备整理的内容

- 图论题里如何快速识别最短路、分层图和最小生成树模型
- DP 题里如何判断状态设计是否过重
- 构造题里如何把直觉变成可验证的方案
